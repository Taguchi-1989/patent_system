# AI自走前提の特許調査支援システム 要件定義書（改訂版）

> **注意**：本書は、特許調査・比較・継続監視を支援するための一般的なシステム要件をまとめたものです。侵害の有無や法的結論を確定するものではなく、最終判断は弁理士・弁護士等の専門家確認を前提とします。

---

## 1. 目的

本システムの目的は、**別システム等から流入する「該当しそうな特許番号リスト」や技術説明**を入力として、**国内外の公開特許を自動取得し、比較表・要約・継続監視レポートまでを極力人手なしで生成すること**です。

特に以下を重視します。

- 候補特許番号の取り込み
- 各国特許庁・公開DBからの公式情報取得
- 請求項分解と比較表生成
- 継続監視・差分抽出
- HTMLベースUIでの確認・合意形成
- NotebookLM 等に流し込みやすい Markdown / HTML / PDF 出力

---

## 2. 前提条件（今回の改訂前提）

### 2.1 開発前提

- 開発は **AIエージェント主導** で進める。
- **GitHub リポジトリを中心** に、Google Colab・CLI・MCP・必要に応じてローカル環境を併用する。
- UI はまず **HTMLベースの軽量プロトタイプ** を複数作成し、合意形成しながら改善する。
- 公開リポジトリには **仕組み・テンプレート・サンプルのみ** を置き、具体案件や評価結果は private / local / closed に分離する。

### 2.2 運用前提

- **ほとんど人が介在しない運用** を前提とする。
- 人が介在するのは以下に限定する。
  - 入力ルール変更
  - 法務・知財の最終レビュー
  - 本番反映承認
- 法的な侵害判断や出願判断は本システムの自動確定対象外とする。

---

## 3. システムのゴール

### 3.1 Phase 1 ゴール（MVP）

- 別システムや手作業で得た **特許番号リスト** を取り込み、対象国ごとに以下を自動取得する。
  - 書誌情報
  - クレーム
  - 明細
  - 経過情報
  - 法的状態
  - ファミリー情報
- 要約と比較表の初稿を自動生成する。

### 3.2 Phase 2 ゴール

- 自社仕様または一般化した技術説明と、対象特許の請求項を比較し、以下の観点整理を自動化する。
  - `MATCH`
  - `MISSING`
  - `UNCLEAR`
- 継続監視による差分抽出を自動化する。

### 3.3 Phase 3 ゴール

- 継続監視結果を蓄積し、以下まで支援範囲を広げる。
  - 競合監視
  - 継続出願・分割監視
  - 出願戦略検討の補助

---

## 4. 対象ユーザー

- **研究開発者・設計者**
  - 自社製品やサービス仕様が、どの請求項要素と重なりそうかを把握したい。
- **知財・技術企画担当**
  - 候補特許の精査、比較表作成、継続監視の初稿をAIに任せたい。
- **個人 / 小規模チーム**
  - 高価な商用特許ツールに依存せず、公開情報とAIで回る調査基盤を持ちたい。

---

## 5. スコープ

### 5.1 実施すること

- **番号リストの取込**
  - CSV / JSON / 他システム出力 / 手入力
- **特許情報取得**
  - JPO / USPTO / EPO を中心
- **要約・請求項分解・比較表生成**
- **継続監視・差分抽出**
- **HTML UI + Markdown / HTML / PDF 出力**
- **NotebookLM 向け出力整形**

### 5.2 実施しないこと

- 法的侵害判断の確定
- FTO意見書の自動作成
- J-PlatPat UI / PATENTSCOPE UI のスクレイピング主導設計
- 機密情報の public repo 掲載

## 6. 機能要件

### 6.1 入力機能

- 特許番号リストを CSV / JSON / テキストで受け取れること。
- 必要に応じて技術説明文も受け取れること。
- 外部システムからの出力フォーマット差異を吸収する正規化処理を持つこと。

### 6.2 取得機能

- 番号ごとに **国 / 庁判別** を行い、適切な取得ルートに振り分けること。
- 日本案件について以下を取得できること。
  - 番号照会
  - 経過情報
  - 引用文献
  - 登録情報
  - 書類
  - permalink
- 米国案件について以下を取得できること。
  - 全文検索結果の参照
  - 特許全文
  - ODPメタデータ
  - 継続関係
  - プロセキューション履歴
- 欧州案件について以下を取得できること。
  - 書誌
  - クレーム
  - 明細
  - ファミリー
  - 法的状態

### 6.3 分析機能

- 対象特許の以下を要約できること。
  - 課題
  - 構成
  - 効果
  - 主要請求項要素
- 請求項を要素分解できること。
- 比較対象仕様と突き合わせて `MATCH / MISSING / UNCLEAR` を付与できること。
- 複数特許の比較表を Markdown / HTML テーブルで出力できること。

### 6.4 監視機能

- 監視対象の番号群について再実行スケジュールを持てること。
- 変更があった案件のみ再取得できること。
- 前回取得との差分を抽出できること。
  - 法的状態
  - 追加書類
  - 継続出願
  - 引用変化
- 差分通知文を自動生成できること。

### 6.5 出力機能

- 以下を案件別に生成できること。
  - サマリ
  - 比較表
  - 差分レポート
  - NotebookLM向けMarkdown
- すべてのレポートに以下の注記を付けること。
  - 「AIによる支援結果であり、最終判断は専門家確認が必要」

---

## 7. 非機能要件

### 7.1 開発運用要件

- public repo に仕組みのみを置き、private data を別管理すること。
- GitHub を唯一の履歴源として扱うこと。
- Colab から GitHub への保存・同期ができること。
- Colab で `pip install` ベースの再現可能なセットアップができること。

### 7.2 品質要件

- すべての分析結果に以下を明記できること。
  - 出典
  - 該当箇所
  - 取得元庁
- 根拠が弱い場合、推測で断定しないこと。
- 取得失敗時には理由を記録し、再試行または代替導線を提示できること。

### 7.3 セキュリティ要件

- 公開特許中心のMVPでは public repo で進める。
- 将来の未公開情報対応を見据え、local / closed 環境へ移植できる構成にする。
- APIキー・認証情報は `.env` 等で外出しし、repoに含めないこと。

## 8. 推奨アーキテクチャ（AI自走前提）

### 8.1 4層構成

1. **Ingestion Layer**
   - 番号リストを受ける層
   - CSV / JSON / 外部出力の正規化
2. **Evidence Retrieval Layer**
   - JPO / USPTO / EPO 公式ソースから原文・書誌・履歴を取得する層
3. **Analysis Layer**
   - 要約エージェント
   - 請求項分解エージェント
   - 比較表エージェント
   - 差分監視エージェント
4. **Presentation Layer**
   - HTML UI
   - Markdown / PDF 出力
   - NotebookLM向けエクスポート

### 8.2 UI 方針

- まずは **HTMLベースの軽量UI** を AI に複数案作らせて合意形成する。
- Colab は実行ハーネスとして使い、最終UIは GitHub Pages または軽量Web App として切り出す。
- 一覧 → 詳細 → 比較表 → 差分履歴 の導線を最小構成で提供する。

---

## 9. AIエージェントの役割定義

### 9.1 黒子エージェント群

- **取得エージェント**
  - 特許番号ごとにデータソースを切り替えて原文を回収する。
- **要約エージェント**
  - 書誌・抄録・請求項・履歴を要約する。
- **比較エージェント**
  - 請求項を要素分解して比較表を作る。
- **監視エージェント**
  - 再取得・差分抽出・通知文生成を行う。
- **UI更新エージェント**
  - HTMLテンプレートの改善、表示部品の修正、スタイル更新を行う。

### 9.2 人が介在するポイント

- 対象番号リストの出所確認
- 比較観点の初期定義
- 例外処理と最終法務レビュー

## 13. 率直な結論

今回の前提、つまり

- UI も AI 主導で HTML ベースで高速に作る
- 黒子的な AI エージェントを継続的に動かす
- Colab / GitHub / CLI / MCP を併用する
- ほとんど人が介在しない

という条件なら、工数は**ゼロからの手作業実装より大幅に圧縮可能**です。

特に、取得層・比較層・運用基盤の主要部品が既に存在するため、

- 取得
- 構造化
- 比較
- 継続監視
- UI改善

を段階的に AI 主導で回せます。
