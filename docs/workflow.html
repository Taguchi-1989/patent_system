決定論パイプライン化と自校（セルフチェック）

この特許調査システムのうち コードで確定できる工程はすべて決定論パイプラインに固め、 判断が要る一点だけを エージェント（黒子）の接続点として残しました。 パイプラインは Python で「形」を宣言し、実行のたびに自分の不変条件（P-NO-GUESS・出典・ 決定性・免責）を 自己検証（自校）します。本書はその設計と、CI／cron への ワークフロー化を説明します。

① 決定論の境界 ② パイプラインの形 ③ 自校ゲート ④ エージェント接続点 ⑤ 実行方法 ⑥ ワークフロー化

01 何を決定論化し、何をエージェントに残したか

原則：確定的な抽出・構造化はコードで、意味の判断だけをエージェントで。 境界をコードの型（Judge プロトコル）で切り、片側を完全に再現可能にしました。

決定論：同じ入力 → 同じ出力（モデル・鍵なし、バイト一致で再現） エージェント：意味判断の継ぎ目（LLM 差し替え可・再現対象外）

DETERMINISTIC コードで確定できる工程

  • 正準化 — 番号文字列 → CanonicalNumber（純標準ライブラリ）。曖昧入力は needs_review。
  • 取得 — PatentSource.fetch()。凍結ソース（Fixture／保存済み BigQuery エクスポート／DL 済み一括 XML）上で決定論的。
  • 抽出要約 — 書誌一行＋請求項のヒューリスティック要素分解。
  • 意味比較（既定） — HeuristicJudge（トークン重なり）。鍵不要・決定論的。
  • 状態・差分 — 内容ハッシュ基準のスナップショット＋フィールド差分。
  • 出力 — Markdown／静的 HTML。免責と出典を必ず付与。

AGENT SEAM 判断をエージェントに委ねる一点

  • 意味判定（LLM Judge） — 技術仕様 × 請求項要素を MATCH / MISSING / UNCLEAR に対応付ける本質的な判断。
  • 差し込み口は compare(spec, summary, judge=…) の Judge プロトコルだけ。他のコードは無変更。
  • LLM の鍵も不要：黒子の Claude Code がそのまま Judge を務める。
  • ただし P-NO-GUESS は構造で強制：ElementVerdict.__post_init__ が根拠なき MATCH を機械的に UNCLEAR へ降格。どの Judge でも捏造判定は不可能。
  • ライブ取得（BigQuery 直クエリ）も非決定論。ゲート前にエクスポートで凍結する。
なぜこの切り方か。 監視・回帰・CI ゲートに必要なのは「同じ入力なら毎回同じ結果」という再現性です。 意味判断は本質的に揺れるため決定論にはできません。そこで揺れる部分を一点に隔離し、 残り全部を自分で検証できる決定論パイプラインに固めました。自校が緑である限り、 エージェントを差し替えても土台の保証（出典・P-NO-GUESS・免責）は崩れません。

形を表すコード（抜粋）

# src/patentkit/pipeline/contract.py — 各 Stage は宣言メタデータ
STAGES = (
  Stage("ingest",    # 1  kind=DETERMINISTIC, gate_ids=("G-NORMALIZE-REVIEW",)),
  Stage("retrieve",  # 2  kind=DETERMINISTIC, gate_ids=("G-PROVENANCE",)),
  Stage("summarize", # 3  kind=DETERMINISTIC),
  Stage("compare",   # 4  kind=AGENT,
        gate_ids=("G-NO-GUESS", "G-EVIDENCE-SUBSTRING", "G-UNCLEAR-REVIEW"),
        agent_seam="compare(spec, summary, judge=<LLM Judge>)"),
  Stage("snapshot",  # 5  kind=DETERMINISTIC, gate_ids=("G-SNAPSHOT-STABLE",)),
  Stage("present",   # 6  kind=DETERMINISTIC, gate_ids=("G-DISCLAIMER",)),
)
Provider パターンが昇格の鍵。 取得は PatentSource 抽象にのみ依存します。 FixtureSource → BigQuerySource に差し替えるだけで、パイプライン本体を 一行も変えずに鍵なしデモから実データへ移行できます（決定論ゲートは凍結エクスポートで担保）。

03 自校ゲート（実行のたびに自分を検証）

各ゲートは、コード内に文書化済みの不変条件を機械的に再検証します。違反しても例外は投げず passed=False と違反項目を返すので、全失敗を一度に提示できます。8 ゲートすべて緑＝ パイプラインが自分の保証を証明した状態です。

ゲート＝1つの不変条件チェック（純関数・個別テスト可能）
ゲート工程証明する不変条件（なぜ要るか）
G-NORMALIZE-REVIEW1 正準化 曖昧な正準化（confidence<0.85 か庁 UNKNOWN）は必ず needs_review。沈黙して誤接合しない（P-NO-GUESS）。
G-PROVENANCE2 取得 全レコードに非空の source。出典なき下流主張をゼロにする（§7.2）。
G-NO-GUESS4 意味比較 根拠スパンなしの MATCH は存在しない。P-NO-GUESS の本丸（構造的に降格）。
G-EVIDENCE-SUBSTRING4 意味比較 全根拠スパンが対象仕様の実在の部分文字列。引用の捏造を不可能にする。
G-UNCLEAR-REVIEW4 意味比較 全 UNCLEAR は needs_review。曖昧を自動確定しない（人へエスカレーション）。
G-SNAPSHOT-STABLE5 状態・差分 同一内容の再保存は「変更なし」。監視が空振り通知を出さない（内容ハッシュ等価）。
G-DISCLAIMER6 出力 出力に必須の免責が含まれる（§6.5）。法的結論の確定でないことを明示。
G-DETERMINISM横断 決定論バックボーンを2回走らせ、成果物がバイト一致。非決定性の混入を検知する自校の本命。

「自校が落ちない」では意味がない。 テスト tests/test_pipeline_contract.py は 各ゲートにわざと壊した入力を与え、ちゃんと FAIL することまで検証します （捏造 MATCH・出典欠落・偽の根拠スパン等）。検知できないゲートは無価値だからです。

04 エージェントの接続点（黒子の差し込み）

意味判定だけを Judge プロトコルの裏に隔離しています。既定は決定論の HeuristicJudge。より賢い判定が要るときは、同じ口に LLM 版を差すだけです。 

# 既定（決定論・鍵なし）: 自校の対象
result = compare(target_spec, summary)                  # judge=HeuristicJudge()

# エージェント差し替え（黒子の Claude Code が Judge を務める。他は無変更）
class AgentJudge:
    def judge(self, element, target_spec, claim_context) -> ElementVerdict:
        # LLM の意味判断。ただし返り値は __post_init__ を通るため
        # 根拠なき MATCH は構造的に UNCLEAR へ降格される（P-NO-GUESS）。
        ...
result = compare(target_spec, summary, judge=AgentJudge())
降格は誰にも止められない。 ElementVerdict.__post_init__ が 「根拠スパンなしの MATCH」「confidence<0.3 の MATCH」を機械的に UNCLEAR に落とします。 つまり どの Judge を差しても、自校ゲート G-NO-GUESS は構造的に守られます。

05 実行方法（鍵ゼロで全部動く）

既定は FixtureSource（ローカル標本・ネットワーク不要）。py で実行します。

# ① 自校：決定論パイプラインを2回走らせ 8 ゲートを検証（CI ゲート / 終了コードで合否）
py scripts/pipeline_selfcheck.py
py scripts/pipeline_selfcheck.py --json        # 機械可読（CI アーティファクト向け）

# ② 通し：正準化→取得→要約→（--spec があれば比較）→ outputs/report.md
py scripts/run_pipeline.py
py scripts/run_pipeline.py --spec samples/target_spec_SAMPLE.md

# ③ 監視：取得→スナップショット→差分→ outputs/diff_report.md
py scripts/monitor.py

# ④ 静的サイト：一覧→詳細→比較→差分履歴 を site/ に生成
py scripts/build_site.py --spec samples/target_spec_SAMPLE.md

# ⑤ 評価：HeuristicJudge の golden 一致率を測定（合否ではなく計測）
py scripts/eval_compare.py

# ⑥ テスト一式（90 件）
py -m pytest tests/ -q
実データへの昇格は --source の差し替えだけ。
--source bq-export --export results.json（BigQuery コンソールの保存 JSON）／ --source bq --project <GCP>（ライブ）／--source bulk --bulk-files …（USPTO 一括）。 決定論ゲートを掛けるときは、ライブではなく凍結エクスポートを入力にしてください。

06 ワークフロー化（CI ＋ 定期監視）

決定論にしたからこそ、機械が回せます。2 つのループに落とし込みます： マージ前ゲート（自校）と 定期監視ループ（cron）。

A. マージ前ゲート — 「土台が壊れていない」を毎回証明

  1. PR / push で pipeline_selfcheck.py を実行。 8 ゲートが緑なら終了コード 0、1つでも落ちれば 1。CI がそのままブロックします。
  2. pytest tests/（90 件）も実行。 純関数・ゲートの失敗検知・回帰しきい値（HeuristicJudge ≥ 0.60）まで担保。
  3. 決定性は fingerprint で監視。 意図せぬ非決定性が混入すると G-DETERMINISM が即落ちます。
# .github/workflows/ に1ジョブ追加するだけ（鍵不要・FixtureSource）
- run: py scripts/pipeline_selfcheck.py     # 自校（合否ゲート）
- run: py -m pytest tests/ -q               # 単体・回帰

B. 定期監視ループ — 既存の cron に同居

監視は既に .github/workflows/monitor.yml が GitHub Actions の cron （毎週水曜 06:00 UTC）で回しています。揮発する Colab ではなく Actions を使うのが設計判断です。

  1. 取得 → スナップショット → 差分。 monitor.py が monitor_state/ に内容ハッシュ付きで保存。
  2. Actions が変更をコミットして戻す。 Git 履歴がそのまま監視の監査証跡になる（§7.1）。差分レポートは artifact。
  3. 実データへは --source 差し替えのみ。 例：--source bulk --bulk-files <週次XML>。パイプライン本体は無変更。
黒子（エージェント）はどこに入るか。 決定論ループが UNCLEAR を出した要素＝ 人／エージェントの確認待ち行列です。Claude Code が Judge として意味判定を補強し、 その結果も再び __post_init__ と自校ゲートを通るので、土台の保証は保たれたまま品質だけ上がります。 人は「リスト出所確認・比較観点定義・最終法務レビュー」に集中します（無人化ではなく確認コスト最小化）。
